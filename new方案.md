# README 中未实现功能的后续实现方案

## 0. 规划前提

当前仓库已经实现的是一套最小可运行闭环：

- FastAPI 提供 3 个接口
- LangGraph 负责 `planner -> executor -> result`
- `PlannerAgent` 和 `ExecutorAgent` 已经分层
- `MockTravelSkill` 提供确定性演示结果
- 会话状态目前只保存在内存里

这份方案的目标不是推翻现有实现，而是在现有骨架上继续迭代，把 README 中明确写着“还没实现”的能力逐步补上。

规划原则如下：

- 不重写现有 FastAPI + LangGraph + Pydantic 骨架
- 先补基础稳定性，再补真实业务能力
- 每一阶段都要有独立可验证结果
- Redis 不作为唯一数据源，数据库才是最终真相来源
- RAG、外部 API、RePlan 都建立在“真实模型调用已经接通”这个前提上

需要先明确一个现实约束：

- README 里提到 `12306`，但火车票数据和下单能力的官方可接入性、合规性、授权方式通常不如地图类 API 清晰
- 所以后续规划里不能默认“必然能直接对接官方 12306”
- 更稳妥的表述应该是：先预留“交通数据服务适配层”，等确认可用且合规的数据源后再落具体供应商

## 1. 建议实现顺序

推荐按下面顺序推进：

1. 先补基础工程能力：数据库持久化、自动化测试底座、超时与幂等保护
2. 再接真实 LLM，让 `Planner` 和 `Executor` 不再依赖固定 mock 逻辑
3. 然后接入第一批真实外部服务，优先做地图 POI、路线、天气这类高价值且较稳定的数据
4. 在已有真实模型和真实工具的前提下，再把工作流升级成支持 RePlan 的多步图
5. 业务链路稳定后，再补 RAG / Milvus，让系统能利用攻略、FAQ、业务知识库
6. 再建设自动化评估体系，用来比较模型、Prompt、RAG、工具策略的改动效果
7. 最后补前端页面，把目前 Swagger 演示升级成真正可操作的产品界面

这样排的原因是：

- 如果没有持久化和测试底座，后面每接一个能力都会放大排查成本
- 如果没有真实 LLM，RePlan 和 RAG 只能停留在空架子层面
- 如果没有真实外部数据，前端再早做也只能包一层 mock 展示

## 2. 分项实现方案

### 2.1 真实模型调用

#### 大致怎么做

- 保留现有 `PlannerAgent` / `ExecutorAgent` 的职责边界
- 在 `planner` 里引入真实 LLM，产出结构化计划，而不是固定返回一个步骤
- 在 `executor` 里保留“技能调用”的主逻辑，但把结果总结和下一步判断交给 LLM
- 增加统一的 Prompt 模板目录，例如 `travel_agent/app/prompts/`
- 所有 LLM 输出继续用 Pydantic 模型校验，避免下游直接消费自由文本
- 配置管理从硬编码升级为环境变量方案，例如 `.env` + `pydantic-settings`

#### 选用技术栈

- LLM SDK：OpenAI 官方 Python SDK 或兼容 OpenAI 协议的模型服务
- 输出约束：Pydantic v2 结构化模型
- 提示词组织：本地 Prompt 模板文件
- 配置管理：`pydantic-settings`

#### 主要优点

- 能直接复用现有 Agent 分层，不需要重写整体架构
- 从 Mock 切到真实推理后，系统才真正具备“智能规划”的起点
- 结构化输出能降低后续接技能和 RePlan 时的脆弱性

#### 主要风险

- 模型输出不稳定，可能出现字段缺失、格式错误、幻觉
- 成本和延迟会上升
- Prompt 需要持续调优，否则 Planner 容易生成不可执行步骤

#### 实现复杂度

- 中

#### 可测试性

- 中
- 可通过“固定输入 -> 校验结构化输出”做契约测试
- 可通过 mock LLM 响应做离线单测
- 还需要少量真实模型 smoke test 验证 Prompt 是否跑通

### 2.2 RAG / Milvus

#### 大致怎么做

- 把攻略、FAQ、城市知识、签证规则、景点说明等文本整理成可入库文档
- 建立离线 ingestion 流程：清洗、切分、打标签、向量化、写入向量库
- 在运行时把 RAG 作为一个独立能力接入，方式有两种：
  - 方式 A：Planner 在规划前先检索知识
  - 方式 B：把检索能力做成一个 Skill，由 Planner 决定何时调用
- 检索结果不直接暴露给用户，而是作为 Planner / Executor 的上下文输入
- 保留文档版本、更新时间、城市标签、来源标记，便于控制数据新鲜度

#### 选用技术栈

- 向量库：Milvus
- Embedding：与主模型同厂或兼容的 embedding 模型
- 文档切分：LangChain text splitter 或自定义规则
- 元数据存储：Milvus metadata + 关系数据库中的文档索引表

#### 主要优点

- 能把旅游知识从代码里解耦出来，后续更新资料不需要频繁改代码
- 可以明显降低纯模型生成带来的幻觉
- 对城市攻略、景点说明、注意事项这类“长文本知识”价值很高

#### 主要风险

- 文档质量和切分质量决定效果，数据脏时 RAG 反而会误导模型
- 向量检索命中不等于答案正确，仍需要 Prompt 和引用策略配合
- Milvus 引入后，部署和运维复杂度会明显上升

#### 实现复杂度

- 高

#### 可测试性

- 中
- 可做“查询 -> 期望命中文档集合”的检索测试
- 可做小规模 gold set 验证回答是否用到了正确知识
- 但最终效果仍然受数据质量和模型行为影响，测试不会像纯接口那样稳定

### 2.3 Redis / 数据库持久化

#### 大致怎么做

- 先把 `InMemoryMemoryStore` 抽象成真正的存储接口
- 引入关系数据库保存会话、执行状态、计划步骤、工具观测、最终结果
- Redis 只承担缓存、锁、幂等键、短时状态，不承担最终持久化
- `POST /sessions/{session_id}/run` 增加幂等控制，避免重复点击导致重复执行
- 若后续运行耗时变长，再考虑把执行任务异步化

#### 选用技术栈

- 主数据库：PostgreSQL
- ORM / 数据访问：SQLAlchemy 2.x
- 迁移：Alembic
- 缓存与锁：Redis

#### 主要优点

- 服务重启后会话不丢，系统才具备可交接和可追溯能力
- 能支撑并发请求、失败恢复和后续多轮对话
- 给评估、审计、回放、问题排查提供真实数据基础

#### 主要风险

- 数据模型设计一旦草率，后续加 RePlan 会频繁改表
- Graph 状态和数据库状态可能出现一致性问题
- Redis 和数据库职责边界如果不清晰，后续维护会变乱

#### 实现复杂度

- 中

#### 可测试性

- 高
- 可以通过集成测试验证创建、运行、查询、重启恢复
- 可以通过并发测试验证锁和幂等是否生效
- 数据模型清晰后，问题复现实验会比内存态更容易

### 2.4 RePlan

#### 大致怎么做

- 现有工作流是固定直线图，后续需要改成带条件分支和循环的状态图
- 增加“判断节点”或“监督节点”，负责判断：
  - 当前目标是否已完成
  - 工具结果是否足够
  - 是否需要重新规划
- Graph 从 `planner -> executor -> result` 升级为：
  - `planner -> executor -> judge -> planner/result`
- 在状态里补充 `step_index`、`remaining_goals`、`error_reason`、`max_iterations` 等字段
- 对循环次数、总预算、总耗时做硬限制，避免死循环

#### 选用技术栈

- 工作流框架：继续使用 LangGraph
- 状态建模：Pydantic + TypedDict/State 类型
- 失败策略：重试、回退、终止状态枚举

#### 主要优点

- 这是从“一次性执行器”升级成“真正 Agent 工作流”的关键一步
- 能在工具失败、数据不足、约束变化时自动调整计划
- 更贴近真实旅游规划场景

#### 主要风险

- 多轮循环后，调试成本会明显上升
- 如果停止条件不清晰，容易出现反复改计划但不收敛
- LLM 成本和执行时长都会增加

#### 实现复杂度

- 高

#### 可测试性

- 中
- 可以用 deterministic mock skill 构造“失败一次后成功”的场景
- 可以测“最大循环次数是否生效”
- 但真实模型参与后，重规划路径会比线性流程更难做到完全稳定复现

### 2.5 真实外部旅游 API

#### 大致怎么做

- 继续沿用当前 Skill 抽象，把每类外部能力都做成单独 Skill
- 第一批建议优先做：
  - 地图 POI 搜索
  - 路线与通行时间估算
  - 天气查询
- 第二批再考虑：
  - 酒店搜索
  - 交通票务数据
  - 图片检索
- 所有第三方返回都先转成系统内部统一结构，避免上层直接依赖供应商字段
- 给每个 Skill 明确输入约定、错误码、超时、重试和降级策略

#### 选用技术栈

- HTTP 调用：`httpx`
- Skill 封装：沿用现有 `SkillRegistry`
- 外部服务：
  - 地图与 POI：高德地图或同类地图服务
  - 天气：稳定公开天气 API
  - 酒店 / 交通：以“已确认合规可接入的供应商”作为前提

#### 主要优点

- 真实外部数据是系统从“演示骨架”走向“可用工具”的关键
- 继续沿用 Skill 边界，可以避免上层工作流被第三方 API 细节污染
- 适合渐进接入，一个 Skill 成熟一个，不必一次接全

#### 主要风险

- 外部 API 配额、鉴权、限流、超时、稳定性都会成为真实问题
- 多个供应商字段风格不同，统一模型会增加适配成本
- `12306` 或酒店直连可能存在商务授权、合规、接口可得性风险

#### 实现复杂度

- 中到高

#### 可测试性

- 中到高
- 可以用录制响应或 mock server 做稳定测试
- 可以为每个 Skill 做契约测试
- 还需要保留一小组真实联调 smoke test，验证 token、配额和真实可用性

### 2.6 自动化评估体系

#### 大致怎么做

- 新建 `evaluation` 模块，沉淀一组固定测试用例
- 用例至少覆盖：
  - 计划结构是否完整
  - 工具选择是否合理
  - 最终答案是否满足预算、城市、天数等硬约束
  - RAG 引用是否命中目标知识
- 区分两类评估：
  - 规则评估：字段完整性、约束满足率、工具调用成功率
  - 主观评估：答案可读性、推荐合理性、路线连贯性
- 先从本地 JSON / YAML case 开始，不急着一开始就接复杂评估平台

#### 选用技术栈

- 用例组织：JSON / YAML / JSONL
- 执行框架：`pytest`
- 指标统计：自定义 evaluation runner
- 可选增强：LangSmith 或类似 tracing/eval 平台

#### 主要优点

- 能把 Prompt、模型、RAG、Skill 的变更变成可比较结果
- 是后续做版本迭代和模型替换的基础
- 比人工“感觉好像还行”更可靠

#### 主要风险

- 评估指标如果设计不好，会出现“分数好看但用户体验不一定更好”
- 用例维护是持续工作，不是一劳永逸
- 主观质量难以完全自动化判定

#### 实现复杂度

- 中

#### 可测试性

- 高
- 这部分本身就是为测试而建
- 规则型指标可稳定回归
- 主观指标可结合人工抽检形成混合验收流程

### 2.7 前端页面

#### 大致怎么做

- 前端第一版不要做成“大而全旅游 App”
- 先做一个单页工作台，只覆盖当前最核心的 3 个动作：
  - 创建会话
  - 触发运行
  - 查看计划、观测和最终结果
- 页面形态建议是左侧输入区、右侧结果区，中间展示当前运行状态
- 等真实外部 API 和 RePlan 稳定后，再考虑对话式 UI、多轮追问、地图视图等增强能力

#### 选用技术栈

- 前端框架：React + TypeScript
- 构建工具：Vite
- 数据请求：TanStack Query
- 本地页面状态：Zustand 或 React state
- 端到端测试：Playwright

#### 主要优点

- Vite + React 轻量，适合当前后端 API 明确、页面不复杂的阶段
- 可以很快把 Swagger 演示升级成业务可理解的演示界面
- 不会因为前端工程过重拖慢后端能力迭代

#### 主要风险

- 如果后续很快要上流式输出、地图组件、复杂权限，再补架构会有返工
- 前端一旦先做“产品感”，团队容易误判后端真实成熟度

#### 实现复杂度

- 中

#### 可测试性

- 高
- 组件层可以做单测
- 页面流程可以做 Playwright E2E
- 对现有 3 个 API 的联调路径也比较明确

### 2.8 自动化测试与工程保障

#### 大致怎么做

- 在 `pytest` 基础上补三层测试：
  - 单元测试：Planner、Executor、Skill、Store
  - 集成测试：FastAPI 路由 + Graph + Store
  - 端到端测试：前端或脚本从创建到查询跑通
- 补齐 README 里目前还没有验证的工程能力：
  - 并发场景
  - 异常恢复
  - 超时控制
  - 性能压测
- 给外部 API Skill 和 LLM 调用加超时、重试、熔断和日志追踪
- 给 `run` 接口增加幂等控制和会话级锁

#### 选用技术栈

- 测试框架：`pytest`
- HTTP 测试：`httpx`
- Mock：`pytest-mock` 或 `respx`
- 性能压测：`k6` 或 `locust`
- 日志与追踪：结构化日志 + request id

#### 主要优点

- 能把“当前只靠 Swagger 和脚本手动验证”的状态升级成可持续回归
- 对后续接真实模型和真实 API 的阶段非常关键
- 一旦出问题，更容易定位是 Prompt、Skill、Store 还是路由层问题

#### 主要风险

- 测试基建容易被低估，前期会占用一部分开发时间
- 如果 mock 过度，测试会和真实行为脱节
- 压测环境和开发环境差异过大时，结果容易失真

#### 实现复杂度

- 中

#### 可测试性

- 高
- 这是最适合分阶段落地、并且最容易持续积累收益的一部分

## 3. 推荐落地节奏

如果按“最小风险、最大可交付性”的思路推进，建议分 4 个阶段落地：

### 第一阶段：把骨架变成可持续开发底座

- PostgreSQL 持久化
- Redis 锁与幂等
- pytest 基础用例
- timeout / retry / logging

阶段目标：

- 服务重启后数据不丢
- 基本接口有自动化回归
- 执行链路有最小稳定性保护

### 第二阶段：把 mock 智能升级成真实智能

- 接入真实 LLM
- Prompt 模板化
- Planner 输出结构化计划
- Executor 结合 LLM 做结果归纳

阶段目标：

- 不改骨架，只把“假智能”换成“真智能”

### 第三阶段：把 mock 数据升级成真实数据

- 接地图/POI/天气
- 统一 Skill 入参与出参
- 引入 RePlan 支持多步任务

阶段目标：

- 系统开始具备真实业务价值，而不是只做演示

### 第四阶段：补知识、评估和界面

- RAG / Milvus
- evaluation 模块
- React 前端

阶段目标：

- 可演示、可比较、可持续优化

## 4. 最终建议

如果只能先选一部分做，不建议优先做前端，也不建议一开始就上 Milvus。

最值得最先投入的三件事是：

1. 数据库持久化 + Redis 锁 + 幂等
2. pytest 自动化测试底座
3. 真实 LLM 接入

原因很简单：

- 这三件事决定后续功能开发是不是可控
- 做完它们，再接外部 API、RePlan、RAG，返工最少
- 如果跳过这三步直接堆页面或堆知识库，系统会很快变得难以维护
