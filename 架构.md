# 智行 AI 旅行助手高并发后端架构方案

## 1. 文档目标

本文档面向当前仓库的旅游规划 Agent 项目，给出一套从“最小可运行闭环”演进到“可支撑高并发访问的生产级后端”的完整方案。设计目标不是追求最复杂的微服务拆分，而是在保持 Python 技术栈优势的前提下，让系统同时具备以下能力：

- 支撑旅游规划、行程生成、RAG 检索、工具调用、多轮对话等核心业务
- 在节假日、热点城市、活动营销等高峰流量下仍能稳定提供服务
- 解决 Agent 项目常见的长链路、不确定性、重复执行、上下文膨胀、外部 API 不稳定等问题
- 能逐步从当前仓库的 `FastAPI + LangGraph + Planner/Executor + Skill + Memory` 原型平滑升级

当前仓库已经具备基本骨架，因此本方案遵循一个原则：**不推翻现有架构，只把现有骨架生产化、服务化、并发化。**

## 2. 项目特点与架构约束

### 2.1 项目业务特点

旅游规划 Agent 与普通 CRUD 后端不同，它具备以下特征：

- 单次请求处理链路长，往往包含 LLM 推理、RAG 检索、多个 Skill 调用和结果整合
- 大部分时间消耗在外部 IO 上，而不是本地 CPU 计算
- 结果天然存在不确定性，需要对模型输出和工具返回做结构化约束
- 单用户一次请求可能触发多次外部调用，因此“单次请求成本”远高于普通接口
- 节假日前后会出现明显流量波峰，且热门城市查询会高度集中

### 2.2 高并发下的核心矛盾

高并发场景下，本项目面对的不是“纯 QPS”问题，而是“长耗时任务在高峰期堆积导致服务雪崩”的问题，主要矛盾包括：

- HTTP 请求线程被长任务占满
- LLM 和外部旅游 API 响应慢，导致尾延迟升高
- 多个相同请求重复进入系统，造成重复推理和重复调用
- 会话状态在多实例部署下不一致
- RAG、对话记忆、工具执行链条过长后，错误会被放大

因此本项目要采用的不是“所有逻辑同步串行跑完再返回”的方案，而是：**同步接入 + 异步执行 + 状态轮询/流式推送 + 多级缓存 + 任务队列削峰 + 状态持久化。**

## 3. 总体架构设计

### 3.1 总体分层

建议系统采用以下分层：

```text
客户端（Web / App / 小程序）
    ↓
Nginx / API Gateway
    ↓
接入层（FastAPI Gateway）
    ↓
会话服务 / 用户服务 / 鉴权服务
    ↓
任务调度层（Celery Worker / Queue / Scheduler）
    ↓
Agent 编排层（LangGraph Workflow Service）
    ↓
├─ Planner Agent
├─ Executor Agent
├─ Guardrail / Judge Node
├─ RePlan Node
├─ Memory Service
├─ RAG Service
└─ Skill Router
    ↓
外部能力层
├─ 地图与路线服务
├─ 酒店服务
├─ 交通票务服务
├─ 天气服务
└─ 图文检索服务
    ↓
数据与基础设施层
├─ MySQL
├─ Redis
├─ Milvus
├─ RabbitMQ / Redis Broker
├─ 对象存储（文档、日志、素材）
├─ Prometheus + Grafana
├─ Loki / ELK
└─ OpenTelemetry + Jaeger/Tempo
```

### 3.2 推荐技术栈

结合你当前项目和 Python 生态，推荐技术栈如下：

- 接入层：`Nginx + FastAPI + Uvicorn/Gunicorn`
- Agent 编排：`LangGraph`
- Agent/Prompt/Tools：`LangChain Core + Pydantic`
- 持久化：`MySQL + SQLAlchemy + Alembic`
- 缓存/锁/限流：`Redis`
- 向量检索：`Milvus + pymilvus`
- 异步任务：`Celery + RabbitMQ`
- 定时任务与补偿：`Celery Beat`
- 流式输出：`SSE` 或 `WebSocket`
- 观测：`Prometheus + Grafana + OpenTelemetry + Jaeger/Tempo`
- 日志：`Loki` 或 `ELK`
- 容器部署：`Docker`
- 高并发弹性扩缩容：`Kubernetes + HPA`

说明：

- 对你这个项目，Python 是主流且合理的选择，因为核心瓶颈在 LLM、RAG、外部 API 的 IO 等待，而不是本地高频数值计算。
- 如果只做课程项目或初期上线，可以先 `Docker Compose`；如果面向高并发正式服务，建议进入 `Kubernetes`。

## 4. 服务拆分与职责设计

### 4.1 API Gateway

API Gateway 的职责不是执行业务，而是统一入口控制。建议放在 Nginx 或 APISIX/Kong 之后的 FastAPI Gateway 中完成：

- JWT 鉴权、用户身份解析
- 接口级限流、黑白名单、签名校验
- 请求 ID 注入，贯穿日志、Trace、任务队列
- 根据接口类型决定同步返回还是异步提交任务
- 聚合部分轻量查询接口

在高并发时，网关层最重要的作用有两个：

- 第一，快速拒绝非法或超额请求，避免把压力传到 Agent 层
- 第二，把长耗时任务改造成异步任务，避免工作线程被占满

### 4.2 会话服务

会话服务负责管理用户的会话生命周期，不负责跑 Agent 主流程。它应保存：

- `session_id`
- `user_id`
- 用户原始 query
- 硬约束信息，如预算、出发地、天数、时间
- 当前任务状态，如 `created/running/finished/failed`
- 最终答案摘要
- 当前绑定的 `task_id`

会话状态落在 MySQL 中，Redis 只缓存热点会话和运行中状态。

### 4.3 Agent 编排服务

该服务是系统核心，建议继续使用当前项目已经采用的 `LangGraph`。它负责：

- 驱动 `Plan -> Retrieve -> Execute -> Observe -> Judge -> RePlan/Finish`
- 控制最大迭代次数、全局超时、降级逻辑
- 维护状态机中的标准上下文
- 将 Planner、Executor、RAG、Memory、Judge 连接为统一流程

这里建议引入两个新节点：

- `JudgeNode`：判断任务是否完成、证据是否足够、是否需要 RePlan
- `GuardrailNode`：对计划和工具入参做合法性校验，拦截越权、幻觉或高风险操作

### 4.4 Planner Agent

Planner 的职责是“做决策”，而不是“直接调用所有工具”。它需要输出结构化计划，至少包括：

- 当前目标
- 子任务列表
- 每步需要的能力类型
- 是否需要 RAG
- 是否需要外部工具
- 是否缺失信息
- 失败后的重规划建议

Planner 输出必须通过 `Pydantic` 模型约束，避免自由文本直接驱动后续系统。

### 4.5 Executor Agent

Executor 的职责是“执行已定计划”。它负责：

- 根据计划选择具体 Skill
- 调用 Skill Router 执行外部服务
- 汇总每一步观察结果
- 给 JudgeNode 提供结构化 observation

Executor 不应直接决定用户目标是否改变，否则系统边界会混乱。

### 4.6 RAG 服务

RAG 必须独立成服务或独立模块，不能散落在 Planner 和 Executor 里。其内部建议分为：

- Query Rewrite：查询改写与补全
- Recall：Milvus 向量召回
- Filter：城市、时间、来源、更新时间过滤
- Rerank：重排
- Assemble：证据组装

RAG 服务对上层只返回结构化知识片段，例如：

- 景点知识卡
- 城市攻略摘要
- 注意事项
- 证据来源列表
- 文档可信度分

### 4.7 Skill Router 与外部能力服务

外部能力不要直接散落在 Agent 节点里，而是统一走 Skill Router。每个 Skill 单独封装：

- 路线规划 Skill
- 酒店搜索 Skill
- 交通数据 Skill
- 天气 Skill
- POI 检索 Skill
- 图片或攻略检索 Skill

每个 Skill 应具备统一的入参与出参、幂等键、超时设置、重试策略和熔断策略。

### 4.8 任务调度层

高并发下，Agent 主流程应异步运行。推荐做法：

- 用户请求到达网关
- 网关写入会话表，生成 `task_id`
- 将任务投递到 `Celery + RabbitMQ`
- Worker 拉取任务后执行 LangGraph 工作流
- 执行中持续刷新 Redis 中的运行态
- 执行完成后写回 MySQL，并通过轮询或 SSE 通知前端

这样可以实现削峰填谷，也可以把 `Planner/RAG/Skill` 的慢调用从 HTTP 请求线程中剥离出来。

## 5. 高并发场景下的核心处理流程

### 5.1 一次典型请求的完整流程

```text
1. 用户提交“从上海去杭州三日游”的请求
2. API Gateway 鉴权、限流、生成 request_id
3. 会话服务写入 MySQL，生成 session_id
4. 网关将执行任务写入 RabbitMQ
5. Celery Worker 消费任务，启动 LangGraph Workflow
6. Planner 生成结构化计划
7. 如需知识补充，调用 RAG 服务检索 Milvus
8. Executor 按计划调用地图、酒店、天气等 Skill
9. JudgeNode 判断证据是否足够，决定 Finish 或 RePlan
10. 结果写回 MySQL，会话状态更新为 finished
11. Redis 中的运行态和热点结果刷新
12. 前端通过轮询或 SSE 获取最终结果
```

### 5.2 为什么必须采用异步化

如果直接在 `POST /run` 中同步执行整个 Agent 流程，会出现以下问题：

- 用户峰值请求会把 Uvicorn worker 长时间占满
- 外部 API 一慢，HTTP 连接就堆积
- 同时运行多个长任务时，整个服务很快进入超时和雪崩

因此推荐接口设计为：

- `POST /sessions`：创建会话
- `POST /sessions/{id}/run`：只负责投递任务，立即返回 `running`
- `GET /sessions/{id}`：查询当前快照
- `GET /sessions/{id}/events`：SSE 流式获取阶段进展

## 6. 高并发支撑策略

### 6.1 服务无状态化

API 层、Agent Worker 层都应尽量无状态化。运行中上下文不保存在本机内存里，而是：

- 热状态放 Redis
- 真正结果和快照放 MySQL
- 检索数据放 Milvus

这样可以实现多实例横向扩容，避免单机内存成为瓶颈。

### 6.2 队列削峰与优先级调度

节假日前夕的热门请求会集中出现，例如“五一杭州三日游”“国庆北京亲子游”。此时不能依赖同步接口硬扛，而应通过消息队列削峰。

建议队列分层：

- 高优先级：付费用户、在线交互任务
- 普通优先级：普通规划任务
- 低优先级：离线评估、报表、知识库构建

Celery 可按不同队列分配不同 worker 池，保证高价值任务不会被批量离线任务挤占。

### 6.3 Redis 在并发中的关键作用

Redis 不只是缓存，还承担多个高并发关键职责：

- 会话热点缓存
- 接口限流
- 分布式锁
- 幂等控制
- 任务运行状态缓存
- 热门目的地攻略缓存
- 熔断后的短时降级缓存

典型用法包括：

- `SETNX` 做任务幂等锁，防止同一会话被重复执行
- 令牌桶/漏桶算法做接口限流
- 热门查询结果缓存，减少重复调用 LLM 和 RAG

### 6.4 数据库持久化与读写分离

MySQL 作为最终真相来源，应负责：

- 用户会话
- 任务表
- 计划表
- 观察结果表
- 最终答案表
- 审计与调用记录

在高并发场景下建议：

- 主库负责写
- 从库负责读
- 热点查询优先走 Redis
- 大字段如原始文档、详细 trace 可转存对象存储，避免主表过肥

### 6.5 RAG 的性能优化

RAG 是 Agent 项目中最容易引入额外延迟的模块之一。高并发下建议：

- 对热门城市和热门景点预生成 embedding
- 按城市、主题对 Milvus collection 或 partition 做拆分
- 对常见查询做 query cache
- 检索结果只返回 top-k 结构化片段，不直接塞入大段原文
- 使用 rerank 过滤低质量召回，减少无效上下文导致的模型浪费

### 6.6 外部 API 的保护策略

地图、天气、酒店、票务等外部 API 是系统最脆弱的部分。高并发下常见问题是：

- 被第三方限流
- 第三方超时
- 偶发错误码或脏数据
- 外部接口返回格式变化

解决思路：

- 每个 Skill 设置独立超时
- 使用指数退避重试，但仅限可重试错误
- 对第三方接入使用熔断器，连续失败后短时打开熔断
- 使用降级缓存返回最近一次可接受结果
- 对外部返回统一做数据校验和标准化

### 6.7 流式输出与用户体验优化

Agent 任务通常需要数秒到数十秒。为避免用户误以为系统卡死，建议：

- 同步返回 `session_id` 和 `running` 状态
- 用 SSE/WebSocket 持续推送阶段事件
- 前端展示“正在规划”“正在检索攻略”“正在调用地图服务”等阶段提示

这样既提升体验，也减少用户因为焦虑而频繁重复提交请求。

## 7. Agent 项目常见问题与解决方案

### 7.1 问题一：模型输出不稳定

表现：

- Planner 输出字段缺失
- tool_name 幻觉
- 参数格式错误

解决方案：

- 所有 Planner/Executor 输出必须绑定 Pydantic Schema
- 增加 GuardrailNode 做二次校验
- 对非法计划直接触发 RePlan 或失败兜底
- 对关键字段设置默认值和白名单校验

### 7.2 问题二：上下文越来越长，性能不断下降

表现：

- 对话越多，token 消耗越大
- 模型开始丢失关键约束
- 响应时间明显升高

解决方案：

- 记忆分层：短期对话、长期事实、硬约束、任务摘要分开存
- 达到阈值后自动摘要压缩
- 硬约束单独结构化存储，不能只依赖摘要文本
- 历史 observation 只保留必要摘要和证据引用

### 7.3 问题三：Agent 死循环或重复 RePlan

表现：

- Planner 和 Executor 来回拉扯
- 一直重新规划但不收敛

解决方案：

- 设置 `max_iterations`
- 设置全局时间预算和 token 预算
- JudgeNode 判断“是否有新增信息”，无新增则禁止继续 RePlan
- 对连续重复计划做循环检测

### 7.4 问题四：重复请求导致重复执行

表现：

- 用户多次点击“开始规划”
- 前端超时重试后再次提交
- 网关重试造成双写

解决方案：

- `session_id + operation_type` 做幂等键
- Redis `SETNX` 加分布式锁
- MySQL 任务表加唯一索引
- 对已运行中的任务直接返回当前状态，而不是重复启动

### 7.5 问题五：外部服务雪崩拖垮主系统

表现：

- 某一个地图/酒店接口超时，整个请求卡住
- 高峰期第三方限流，导致错误率上升

解决方案：

- Skill 级超时
- 熔断 + 降级
- 本地缓存最近结果
- 将第三方错误包装成标准 observation，交给 JudgeNode 决定是否换路或降级回答

### 7.6 问题六：RAG 结果过时或不可信

表现：

- 检索到陈旧攻略
- 召回内容与目标城市不匹配

解决方案：

- 文档入库时写入更新时间、城市标签、来源类型
- 检索后按时间和地域过滤
- 引入 rerank 与 source score
- 回答中保留 evidence_refs，支持审计和溯源

### 7.7 问题七：高峰期成本失控

表现：

- 模型调用量暴涨
- 相似请求反复消耗同样的 token

解决方案：

- 热门查询结果缓存
- 简单请求优先走轻量模型
- 复杂任务再升级到高质量模型
- 建立用户级、租户级成本预算控制
- 对低价值批量请求做排队和配额限制

## 8. 中间件选型与作用说明

### 8.1 Nginx / APISIX / Kong

用于统一入口、反向代理、TLS 终止、基础限流、灰度发布。若系统后期服务增多，建议引入 APISIX 或 Kong 管理网关策略。

### 8.2 Redis

用于缓存、限流、锁、幂等控制、会话热状态、热点结果缓存，是高并发场景最关键的中间件之一。

### 8.3 RabbitMQ

用于 Celery 的 Broker，承担任务排队、削峰、失败重试和优先级分发。相比直接同步执行，RabbitMQ 能显著提升峰值期稳定性。

### 8.4 MySQL

作为最终事务数据存储，保存用户、会话、任务、计划、执行结果、审计日志等关键数据。

### 8.5 Milvus

用于旅游知识、攻略、FAQ、景点资料等向量化检索，是 RAG 体系的核心中间件。

### 8.6 Prometheus + Grafana

用于采集和展示：

- QPS
- 接口 RT
- Worker 队列积压
- LLM 调用耗时
- Skill 成功率
- RAG 检索耗时
- 任务成功率

### 8.7 OpenTelemetry + Jaeger/Tempo

用于分布式链路追踪，能把一次请求从网关、队列、Agent、RAG、Skill 一路串起来，是排查 Agent 长链路问题的核心工具。

### 8.8 Loki / ELK

用于集中日志管理。Agent 系统调试很依赖日志，建议每一步都打印结构化日志，包含：

- request_id
- session_id
- task_id
- node_name
- tool_name
- model_name
- latency_ms
- success/failure

## 9. 监控指标设计

为了让系统在高并发下可运维，至少要监控以下指标：

- API 层：QPS、P95/P99、错误率、限流次数
- 队列层：任务堆积长度、消费速率、重试次数、死信数量
- Agent 层：平均迭代次数、RePlan 比例、完成率、失败率
- RAG 层：召回耗时、命中率、重排耗时
- Skill 层：各第三方接口成功率、超时率、熔断次数
- 存储层：MySQL 慢查询、Redis 命中率、Milvus 检索延迟
- 成本层：单任务 token 消耗、单用户成本、缓存节省比例

## 10. 典型故障与降级策略

高并发场景下，系统不能假设所有组件永远正常，因此必须设计“局部失败但整体可用”的能力。

### 10.1 LLM 服务异常

可能问题：

- 上游模型接口超时
- 模型服务限流
- 模型输出质量突然下降

解决方案：

- 配置主模型和备用模型
- Planner 和 Executor 分别允许不同降级模型
- 对简单查询切换到轻量模型
- 连续失败后熔断一段时间，避免无效重试

### 10.2 队列堆积

可能问题：

- 节假日请求激增，RabbitMQ 积压严重
- Worker 消费速度低于入队速度

解决方案：

- 按优先级拆分队列
- 按队列长度自动扩容 Worker
- 为离线任务设置独立队列，避免挤占在线任务
- 超时未消费任务进入告警或死信队列

### 10.3 Redis 异常

可能问题：

- 热缓存丢失
- 分布式锁失效
- 限流策略不可用

解决方案：

- Redis 采用哨兵或集群部署
- 锁与幂等控制同时保留 MySQL 唯一约束兜底
- Redis 不可用时降级为数据库查询，但限制高成本接口流量

### 10.4 MySQL 压力过大

可能问题：

- 慢查询变多
- 会话表和结果表膨胀

解决方案：

- 读写分离
- 大字段拆表或归档到对象存储
- 热点状态优先走 Redis
- 定期归档历史任务与 trace

### 10.5 Milvus 或 RAG 服务异常

可能问题：

- 向量检索超时
- 知识服务不可用

解决方案：

- RAG 设置独立超时，超时后允许走“无 RAG 降级回答”
- 对热门知识结果做缓存
- JudgeNode 识别“证据不足但可回答”的场景，输出带免责声明的基础方案

### 10.6 外部 Skill 服务异常

可能问题：

- 酒店或地图服务被限流
- 某一供应商接口不可用

解决方案：

- 多供应商适配层
- 熔断后切换备用服务
- 无法获取精确结果时退化为“推荐级方案”而非“预订级方案”

## 11. 部署演进建议

### 11.1 第一阶段：当前项目升级版

适合课程项目、毕设初版、原型验证。

- 单体 FastAPI
- 单个 LangGraph 服务
- MySQL + Redis
- Celery + RabbitMQ
- Milvus 可后补
- Docker Compose 部署

### 11.2 第二阶段：中等并发生产版

适合小规模线上服务。

- 网关与 Agent Worker 分离部署
- 多个 FastAPI 副本
- 多个 Celery Worker 副本
- Redis 哨兵或集群
- MySQL 主从
- Milvus 独立部署
- Prometheus + Grafana + Loki

### 11.3 第三阶段：高并发正式版

适合节假日高峰和持续在线业务。

- Kubernetes 管理所有服务
- HPA 根据 CPU、队列长度、RT 自动扩容
- 网关、Agent、RAG、Skill 分服务部署
- Redis Cluster、MySQL 高可用、Milvus 集群化
- 引入灰度发布、熔断策略、故障演练和压测体系

## 12. 与当前仓库的落地衔接

你当前仓库已经有：

- FastAPI 接口层
- LangGraph 工作流
- Planner / Executor 分层
- Skill 抽象
- MemoryStore

因此建议按以下顺序演进：

1. 把 `InMemoryMemoryStore` 抽象成接口，先接 MySQL + Redis
2. 把 `POST /run` 改成异步投递任务，而不是同步执行
3. 增加任务表、执行状态表、观察结果表
4. 新建 `rag/` 模块，独立承载检索服务
5. 用真实 Skill 替换 `mock_travel`
6. 在 LangGraph 中增加 `JudgeNode` 和 `GuardrailNode`
7. 补齐日志、监控、链路追踪和限流

这条路线与当前代码兼容，不需要重写整个项目。

## 13. 最终结论

你的旅游规划 Agent 项目完全适合采用纯 Python 技术栈实现高并发后端架构。其关键不在于更换语言，而在于是否采用了正确的工程化手段：**异步任务队列、状态持久化、缓存与限流、RAG 服务解耦、Skill 标准化、Agent 工作流约束、监控追踪与降级保护。**

如果按照本文方案演进，系统可以从当前最小闭环逐步升级为一套具备以下特征的生产级 Agent 后端：

- 前台高并发接入稳定
- 后台长链路 Agent 任务可控
- 外部 API 波动可降级
- 会话状态一致且可恢复
- RAG 和记忆可扩展
- 整体流程可观测、可审计、可迭代

简而言之，这个项目不是不能做高并发，而是必须用“面向 Agent 的服务化架构”来做高并发。只要把同步串行改造成异步编排，把状态和风险控制做好，Python 完全可以支撑你的智行 AI 旅行助手进入可上线、可扩展、可持续演进的阶段。
