# 系统架构

## 1. 架构定位

基于已选定的**方案二：双层多 Agent 协作**，系统采用“LangGraph 状态图编排 + LangChain 能力封装 + 规划 Agent + 执行 Agent + 共享记忆层 + 工具层”的骨架。

目标不是先做完整产品，而是先形成一套**足够支撑编码启动**的工程结构，确保以下主线能跑通：

- 感知 -> 规划 -> 执行 -> 观察 -> RePlan
- RAG 检索增强
- 路线规划 Skill 与外部旅游服务调用
- 图文相关检索
- 过程与结果评估

当前明确不展开：提示词细节、模型调参、召回排序算法细节、数据库表级实现。

## 2. 模块拆分

建议拆为 8 个一级模块：

1. `api`：统一对外接口层，承接 App/Web/CLI 请求
2. `graph`：基于 LangGraph 的状态图编排层，驱动双 Agent 主循环
3. `agents`：规划 Agent 与执行 Agent，基于 LangChain Agent/Chain 组织
4. `memory`：共享记忆、上下文分层、摘要压缩
5. `rag`：基于 LangChain Retriever 的查询改写、召回、重排、知识拼装
6. `skills`：路线规划、酒店、火车票、图片检索等工具能力
7. `evaluation`：自动化评估与运行观测
8. `infra`：模型接入、Milvus、缓存、配置、权限控制

## 4. 各模块职责

### 4.1 api

- 接收用户输入、会话参数、约束条件
- 返回结果、状态、执行过程摘要
- 不直接承载规划逻辑，只做协议转换和入参校验

### 4.2 graph

- 维护一次任务会话的主状态
- 基于 LangGraph 驱动 `Plan -> Execute -> Observe -> RePlan/Finish`
- 控制最大循环次数、超时、失败恢复、降级策略
- 负责在 Planner 和 Executor 之间传递标准化状态
- 显式定义节点、边、条件跳转和终止条件

### 4.3 agents

`PlannerAgent`
- 解析用户目标与约束
- 生成任务计划、子任务列表、缺口信息
- 判断是否需要检索、是否需要调用 Skill、是否需要 RePlan
- 优先封装为 LangChain Chain / AgentExecutor，可被 LangGraph 节点调用

`ExecutorAgent`
- 根据计划选择 Skill / RAG / 外部服务
- 执行工具调用并回传结构化观察结果
- 不改写用户目标，只负责落实当前计划步骤
- 通过 LangChain Tools 统一调度外部能力

### 4.4 memory

- 维护共享记忆层
- 管理短期对话上下文、任务状态、关键事实、已完成动作
- 在上下文过长时做摘要压缩，但保留硬约束与关键决策

### 4.5 rag

- 对查询做改写与补全
- 向 Milvus 发起召回
- 对召回结果重排和证据组装
- 向 Planner/Executor 返回结构化知识片段，而不是原始文档堆叠
- 优先复用 LangChain Retriever / VectorStore 抽象，减少底层胶水代码

### 4.6 skills

- 抽象全部可执行能力
- 统一入参与出参，隐藏外部 API 差异
- 路线规划、酒店、12306、图片检索都作为独立 Skill 注册
- 对上层暴露为 LangChain Tool，便于 ExecutorAgent 和 LangGraph 节点复用

### 4.7 evaluation

- 对单模块和端到端流程做自动评估
- 输出任务成功率、工具调用成功率、计划稳定性、RAG 命中有效性、响应时延等指标
- 复用真实案例或合成案例做回归

### 4.8 infra

- 统一模型与 Embedding 接入
- 管理向量库、缓存、日志、配置、鉴权、限流
- 提供权限校验与外部服务访问边界

## 5. 核心接口或 API

先定义内部接口，再决定 HTTP 形式。下面是建议的稳定接口。

### 5.1 对外 API

#### `POST /sessions`

创建一次旅行任务会话。

请求体：

```json
{
  "user_id": "u_001",
  "query": "五一从上海去杭州玩三天，预算3000，想看西湖和灵隐寺",
  "constraints": {
    "budget": 3000,
    "start_city": "上海",
    "travel_days": 3
  }
}
```

响应体：

```json
{
  "session_id": "sess_xxx",
  "status": "created"
}
```

#### `POST /sessions/{session_id}/run`

启动主链路执行。

响应体：

```json
{
  "session_id": "sess_xxx",
  "status": "running"
}
```

#### `GET /sessions/{session_id}`

查询当前任务状态、阶段结果、最终方案。

## 7. 主链路时序

```text
User/API
  -> API.create_session
  -> MemoryStore.load_context
  -> LangGraph.start
  -> planner_node
  -> RAGService.retrieve (按需)
  -> planner_node.update_plan
  -> executor_node
  -> LangChain Tool.invoke / External API
  -> executor_node.collect_observations
  -> MemoryStore.save_event
  -> router_node.check_finish
  -> planner_node.replan (如未完成)
  -> ...循环...
  -> result_node.output
  -> EvaluationRunner.record_trace
```

建议的控制规则：

- 每次任务最多 `N` 轮 RePlan，默认建议 3
- 执行失败时先局部重试，再触发 RePlan
- Planner 只基于结构化观察结果重规划，不直接读取外部原始响应
- LangGraph 只负责编排状态流转，不承载复杂业务细节
- 最终输出前必须经过结果组装层，避免把中间噪音直接返回给用户

## 8. 权限、隔离和边界

### 8.1 权限分层

- `api` 层：用户身份、请求合法性、频控
- `graph` 层：仅能调内部节点与服务，不直接持有外部平台密钥
- `skills` 层：按 Skill 持有最小化访问凭证
- `infra/security`：统一做密钥管理、审计日志、敏感字段脱敏

### 8.2 隔离原则

- Planner 与 Executor 逻辑隔离，避免“又规划又执行”
- LangGraph 编排层与 LangChain Tool/Chain 实现隔离，避免流程和能力混写
- RAG 只提供证据，不直接下决策
- Skill 只返回事实与结构化结果，不生成面向用户的长文本结论
- Memory 负责保存与压缩，不负责业务判断

### 8.3 外部边界

- 高德地图、酒店、12306、图片检索服务都通过 `skills/*_client.py` 接入
- 外部服务失败不能直接打断全局流程，需要映射为标准错误码
- 向量库 Milvus 仅用于知识/图像向量检索，不承载事务状态
- LangChain 负责统一抽象，不应反向侵入底层客户端协议

### 8.4 当前不稳边界

以下边界当前仍需在编码前确认：

- 酒店与 12306 是否接真实生产 API，还是先做 Mock/沙箱适配
- 图像检索的数据源范围，是本地知识库、开放图库，还是旅游 POI 图片库
- 用户画像是否进入长期记忆；若进入，需要明确隐私与保存周期
- “共享记忆层”最终落 Redis、数据库还是文件/对象存储，目前未定
- 评估集来源未定：人工构造、真实问答日志、还是混合方式
- LangChain 版本与 LangGraph 版本要先锁定，避免接口变动影响骨架稳定性

这些项不影响骨架开工，但会影响二阶段实现深度。

## 9. 技术栈选择

在题目硬约束下，并结合你指定的框架路线，建议如下：

- 语言：Python 3.9
- Web/API：FastAPI
- Agent 能力层：LangChain
- 状态编排层：LangGraph
- 数据校验：Pydantic 1.x 或与 LangChain/LangGraph 兼容版本
- 模型接入：LangChain ChatModel 封装 + 自定义模型工厂
- 向量库：Milvus
- 向量检索抽象：LangChain VectorStore / Retriever
- 缓存/短状态：Redis
- 关系型存储：PostgreSQL 或 SQLite（开发期）
- 异步任务：Celery 或轻量 BackgroundTasks（二期再升级）
- 测试：pytest
- 可观测：LangSmith（可选） + 结构化日志 + Trace ID + 评估报告

选择原则：

- LangChain 负责模型、Prompt、Tool、Retriever 抽象
- LangGraph 负责多 Agent 状态流转、条件分支和 RePlan 闭环
- 业务接口层仍保持自定义契约，避免直接把 LangChain/LangGraph 类型暴露到全系统
- Python 3.9 下优先选择兼容版本组合，避免为了新语法牺牲依赖稳定性

## 10. 编码起步建议

第一阶段先把以下骨架做出来即可：

1. `api + graph/state + graph/workflow + planner/executor contracts`
2. `memory` 的最小可用上下文存取
3. `skills/base.py`、`skills/registry.py` 与 `LangChain Tool Adapter`
4. `rag/retriever.py` 的占位接口，先打通 LangChain Retriever 到 Milvus
5. 一条可跑通的伪链路：用户请求 -> LangGraph planner_node -> executor_node -> Mock Tool -> 返回结果

做到这一步，就已经具备继续展开编码的稳定基础。
